What QuickProbe inspects

Comprehensive health metrics via WinRM and SSH for mixed fleets, keeping CPU, memory, disks, services, network, and system events visible.

Windows hosts (WinRM)

  • OS info, disks with alert thresholds (default 10% free), and critical service status (defaults to WinRM).
  • CPU load (normalized by core count), total/used memory, process count, and high-CPU process detection (>90%).
  • Uptime hours, pending reboot signals (Windows Update, Component-Based Servicing, etc.).
  • WinRM listeners, firewall profiles (Domain/Private/Public), and network adapters (multiple fallbacks).
  • Recent critical/error events from System/Application logs (last 30 minutes, up to 10 entries).
  • Reachability via ping + TCP (ports 3389/5985 by default, ~1.2s timeout per port, configurable per host).
  • Quick probe heartbeat (~20s intervals, 30s per-host cooldown on manual triggers intervals with 500ms jitter, 3-minute stale threshold) with uptime, CPU, memory, processes, and service status.

Linux hosts (SSH)

  • OS info from /etc/os-release, disks from df, and service status via systemctl (password auth).
  • CPU load from /proc/loadavg, memory from /proc/meminfo, and process count.
  • Uptime hours from /proc/uptime and top CPU processes from ps.
  • Reachability via ping + TCP (SSH port 22 by default, configurable per host).
  • Quick probe heartbeat (~20s intervals) reuses full probe path (system_health_probe) for core metrics.
  • Windows-only extras (WinRM listeners, firewall, reboot signals, recent events, network adapters) are skipped on SSH targets.

Storage & deployment

QuickProbe uses SQLite for persistence and runs locally; share access by hosting it on a Windows machine and exposing it via RemoteApp/RDP.

Local mode (default)

  • SQLite database in %APPDATA%\QuickProbe\quickprobe.db (created on demand).
  • Hosts, settings, and view state stored in SQLite; dashboard cache is local-only JSON.
  • Credentials stored in Windows Credential Manager (DPAPI); no cloud storage.
  • Encrypted backup/restore with AES encryption; pre-restore snapshots for safety.
  • Legacy CSV import on first run; one-time migration to SQLite with timestamped backup.

RemoteApp/RDP

  • Multiple Windows sessions under the same account share one SQLite database.
  • SQLite WAL (Write-Ahead Logging) allows concurrent reads; writes serialize with 5s timeout.
  • Each Windows account has its own %APPDATA%\QuickProbe (isolated data).
  • Simultaneous writes resolve to the last committed transaction; no merge/conflict prompts.

Security & privacy

  • All credentials stored in Windows Credential Manager; no secrets logged or sent to external services.
  • Telemetry is not collected or sent to cloud services.
  • Backups are AES-encrypted; password required for restore.

Technical details

Built on Tauri (Rust backend, web UI) with SQLite persistence, WinRM/SSH probes, and Windows Credential Manager integration.

Component Technology
Backend Rust (Tauri framework), SQLite with WAL mode
Frontend Static HTML/CSS/JavaScript (dashboard-all.html, hosts.html, options.html)
Windows probes WinRM (PowerShell remoting via Invoke-Command)
Linux probes SSH (password auth) via ssh2 crate
Credentials Windows Credential Manager (DPAPI) via windows-credentials crate
Storage SQLite (hosts, settings, kv tables); local JSON cache for dashboard
Backup/Restore AES-encrypted ZIP (manifest.json + quickprobe-backup.json)

Requirements

  • Client: Windows 10/11 or Windows Server 2016+; PowerShell 5.1+
  • Windows targets: WinRM enabled (Enable-PSRemoting -Force); network access to ports 5985/5986
  • Linux targets: SSH reachable with password auth (port 22 or custom)
